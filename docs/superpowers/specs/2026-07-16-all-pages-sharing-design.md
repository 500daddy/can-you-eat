# 全页面分享能力设计

日期：2026-07-16

## 目标

让 `app.json` 注册的每个页面都能从微信右上角菜单“转发给好友”，同时避免把私有记录标识、编辑上下文、登录来源或本地图片路径发送给接收者。

对于“分享到朋友圈”，只在无需私有参数、能够独立打开的安全页面开放。微信朋友圈分享固定落在当前页面，回调不能像好友转发一样改写目标路径，因此不在操作页强行开启。

## 方案比较

### 方案 A：每个页面显式接入共享工具（采用）

新增一个分享工具，集中生成标题、好友分享目标和朋友圈参数。每个页面显式引入并展开分享处理器，家庭共享页保留已有的邀请卡覆盖逻辑。

优点：符合微信页面级分享模型，页面能力一目了然，工具和策略可单测。缺点：需要机械修改所有页面文件。

### 方案 B：全局改写 `Page` 构造器

在运行时包装微信全局 `Page`，自动注入分享函数。改动文件少，但依赖非业务层的全局劫持，调试困难，也可能与微信运行时或测试桩冲突，因此不采用。

### 方案 C：在每个页面重复实现回调

不增加共享工具，逐页复制 `onShareAppMessage` 和 `onShareTimeline`。实现直接，但会产生大量重复代码，后续修改标题或安全策略容易遗漏，因此不采用。

## 分享策略

### 好友转发

所有已注册页面都定义 `onShareAppMessage`。

- 安全浏览页分享当前页面；只保留该路由明确允许的查询字段。
- 私有记录页、编辑页和设置页分享统一首页 `/pages/index/index`。
- 家庭共享页继续使用现有邀请标题和 `inviteId` 路径，确保邀请流程不回退。
- 未能获得当前路由、路由不在允许列表或参数异常时，安全回退首页。

当前页分享白名单：

- `pages/index/index`
- `pages/food/search`，仅允许 `keyword`
- `pages/food/name-search`，仅允许 `keyword`
- `pages/purchase-plan/index`
- `pages/quick-process/index`，不转发私有记录参数 `id`
- `pages/recognize/index`
- `pages/reminder/index`
- `pages/mine/index`
- `pages/feedback/index`，不转发表单预填参数
- `pages/about/index`

统一回退首页的页面：

- `pages/recognition-log/index`
- `pages/food/detail`
- `pages/food/add`
- `pages/food/edit`
- `pages/settings/account`
- `pages/settings/baby`
- `pages/avatar-crop/index`
- `pages/settings/reminder`
- `pages/family/member`

`pages/family/index` 使用专用邀请分享，不进入通用目标策略。

### 朋友圈分享

朋友圈只在上述安全浏览页定义 `onShareTimeline`。朋友圈查询参数使用与好友转发相同的字段白名单；私有页和操作页不定义该回调，避免打开缺少上下文或泄露标识的页面。

家庭共享页不开放朋友圈邀请。家庭邀请是点对点能力，并且当前邀请 ID 保存在页面数据中，不适合作为公开朋友圈入口。

## 代码结构

新增 `utils/share.js`，职责如下：

- 维护默认分享标题、首页回退路径和路由策略。
- 从当前页面栈读取路由与启动参数。
- 按字段白名单编码查询字符串。
- 生成通用 `onShareAppMessage` 和可选的 `onShareTimeline` 处理器。
- 导出纯函数，便于在 Node 测试中验证路径和参数安全规则。

页面只负责声明是否使用通用处理器。家庭页在通用处理器之后声明现有 `onShareAppMessage`，以页面专用逻辑覆盖通用好友转发。

## 错误与隐私处理

- 查询值统一使用 `encodeURIComponent`，忽略 `undefined`、`null` 和空字符串。
- 不转发食材记录 ID、成员信息、登录来源、反馈正文和本地图片地址。
- 微信页面栈不可用或返回异常时，不抛错，好友转发回退首页。
- 分享功能不读取或修改用户数据，不引入云函数调用。

## 测试与验收

先增加失败测试，再实现代码：

1. 遍历 `app.json.pages`，确认每个页面的页面定义都具备好友转发处理器。
2. 验证安全浏览页生成当前路径，并且只保留允许的查询字段。
3. 验证私有页和异常上下文回退首页。
4. 验证朋友圈只出现在安全浏览页，查询参数经过同一白名单。
5. 运行现有完整 Node 测试集，确认家庭邀请分享的标题和路径保持不变。

真机验收：在首页、安全浏览页、食材详情页、编辑页和家庭共享页分别打开右上角菜单；好友转发均可用，操作页分享卡打开首页，家庭邀请卡打开带 `inviteId` 的家庭页，朋友圈只在安全浏览页可用。
