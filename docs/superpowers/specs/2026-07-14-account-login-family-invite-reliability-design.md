# 家长登录与家庭邀请可靠性优化设计

日期：2026-07-14

## 背景

家长账号和家庭共享已经具备基础页面与云函数，但联调中暴露出六类问题：

- `user_profiles` 集合缺失时，登录失败且用户无法判断原因。
- 云环境 ID 的查找和客户端配置路径不清楚。
- 微信头像展示比例异常，昵称选择反馈不明确，登录等待时间过长。
- 登录完成后把食材同步失败笼统描述为网络问题。
- 家庭邀请依赖手工输入邀请码，不适合老人使用。
- 账号和家庭的归属边界不够明确，存在重复加入家庭的风险。

本次优化在现有家长账号、家庭权限和食材合并能力上补齐可靠性，不重新设计宝宝档案和食材主流程。

## 已确认决策

采用“个人家庭可被替换，正式家庭锁定”的规则：

- 新账号自动创建的单人空间是个人占位家庭。
- 个人家庭可以接受第一次家庭邀请。
- 首次创建邀请、成功加入其他家庭或家庭出现第二位成员时，家庭升级为正式家庭。
- 正式家庭中的账号不能加入其他家庭。
- 微信分享卡片是主要邀请方式，邀请码降级为备用方式。

## 目标

- 家长登录尽快完成，不被头像上传和食材同步阻塞。
- 头像、微信昵称和登录进度有清楚、稳定的交互反馈。
- 同步失败时保留本机数据，并展示符合真实原因的用户文案。
- 家人通过微信分享卡片直接进入邀请确认页。
- 前后端共同保证一个账号只能属于一个正式家庭。
- 开发者能快速区分集合、部署、环境和网络问题。

## 非目标

本次不增加：

- 多家庭切换。
- 主动退出正式家庭后加入另一个家庭。
- 家庭之间的数据搬迁工具。
- 手机号、邮箱或密码登录。
- 自动读取未经用户确认的微信头像或昵称。
- 家庭聊天、动态和消息中心。

## 家庭状态模型

### 个人家庭

新账号首次建立家庭时写入：

- `kind: personal`
- `formalizedAt: null`
- `formalizedReason: null`

个人家庭只允许一个有效成员，创建者即当前账号。它用于承载用户登录前后的本机食材合并，不代表用户主动组建了正式家庭。

### 正式家庭

发生以下任一事件时，将家庭更新为 `kind: formal`：

- 创建者或管理员成功创建一条家庭邀请。
- 第二位成员加入。
- 用户接受邀请并进入其他家庭。

同时记录：

- `formalizedAt`
- `formalizedReason: invite_created | member_joined | invite_accepted`

正式家庭即使之后只剩一个成员，也不会退回个人家庭。

### 旧数据兼容

缺少 `kind` 字段的家庭按以下顺序判定：

1. 有两位及以上有效成员，判为正式家庭。
2. 有有效或已使用邀请、成员加入记录，判为正式家庭。
3. 当前账号不是创建者，判为正式家庭。
4. 其余只有创建者一人的自动家庭，补记为个人家庭。

判定完成后写回字段，后续不再重复推断。

## 家长登录交互

### 头像

- 头像按钮固定为正方形，不受微信原生 `button` 默认内边距影响。
- 预览使用等宽高容器和 `object-fit: cover` 等效展示。
- 继续使用微信小程序的 `chooseAvatar` 主动选择能力。
- 用户选中头像后立即显示本机预览，上传失败不清空已选内容。

### 昵称

- 输入框继续使用 `type="nickname"`。
- 同时处理 `input`、`change` 和 `blur`，微信昵称建议被选中后立即回填。
- 输入框下只保留简短说明：“可选择微信昵称，也可以自己填写”。
- 昵称为空时禁止提交，并聚焦输入框；不再使用“未登录”等占位昵称。

### 登录阶段

按钮按真实阶段展示：

1. `正在登录`：调用登录云函数并建立账号身份。
2. `登录成功`：保存账号会话并返回“我的”页。
3. 后台任务：上传头像、初始化家庭、合并本机食材和刷新家庭摘要。

建立账号会话所需的最小成功条件是：

- 登录云函数返回当前 OpenID。
- 家长资料接口可以创建或读取账号资料。

家庭初始化或食材同步失败不撤销登录状态。此时继续使用完整本机食材库，并在账号卡片显示可重试状态。

## 后台同步

### 执行方式

- 账号资料保存成功后立即落地本机会话并返回上一页。
- 头像上传、家庭初始化可以并行执行。
- 获得有效家庭后再执行本机食材合并。
- 页面 `onShow` 时恢复未完成任务，但同一任务只允许一个进行中的请求。
- 每个步骤保持幂等，重复点击“重新同步”不会生成重复食材或成员记录。

### 用户状态

账号卡片只展示用户能采取行动的状态：

- `同步中`：后台正在上传资料或同步食材。
- `部分食材尚未同步`：本机数据仍然完整，可点击“重新同步”。
- `家庭信息暂不可用`：账号已登录，本机食材仍可使用。

不再把所有失败都描述为“网络有问题”。开发日志保留结构化原因码：

- `COLLECTION_MISSING`
- `FUNCTION_NOT_DEPLOYED`
- `ENV_MISMATCH`
- `PERMISSION_DENIED`
- `NETWORK_ERROR`
- `SYNC_CONFLICT`

普通用户只看到可理解的状态和重试入口，技术原因留在开发者工具及云函数日志中。

## 微信分享邀请

### 创建邀请

- 家庭页的主按钮改为微信分享按钮，使用小程序原生分享能力。
- 分享前调用 `familyApi.createInvite` 生成一次性邀请标识。
- 邀请默认 7 天有效，只能成功使用一次，可被邀请人撤销。
- 首次成功创建邀请时，个人家庭立即升级为正式家庭。
- 分享路径为 `/pages/family/index?inviteId=<opaque-token>`。
- 链接中不包含 OpenID、家庭名称或其他可读个人信息。

### 打开邀请

邀请页根据账号状态处理：

- 未登录：暂存 `inviteId`，完成家长账号登录后自动返回邀请确认页。
- 已登录且只有个人家庭：展示家庭名称、邀请人和确认加入按钮。
- 已在目标家庭：直接进入家庭页，不重复创建成员记录。
- 已在其他正式家庭：禁止加入，并提示“你已加入一个家庭，暂时无法加入其他家庭”。
- 邀请过期、已使用或被撤销：提示联系邀请人重新发送。

邀请码入口保留在“其他加入方式”中，与微信分享使用同一条后端加入逻辑。

### 确认加入

`familyApi.acceptInvite` 在服务端按顺序校验：

1. 邀请存在、有效、未使用且未撤销。
2. 当前账号是否已经是目标家庭成员。
3. 当前账号是否属于其他正式家庭。
4. 当前个人家庭是否仍满足可替换条件。
5. 写入目标家庭成员、迁移个人家庭记录、归档个人家庭并消费邀请。

重复调用返回同一结果，不重复迁移数据。关键写入放入数据库事务；如果当前云数据库能力无法覆盖完整事务，则使用确定性成员键和迁移状态字段保证可恢复、可重试。

## 数据迁移

个人家庭接受邀请时迁移：

- `user_food_records`
- 尚未完成的采购计划
- 需要保留的识别记录
- 家庭宝宝设置仅在目标家庭未设置时迁移

不迁移：

- 旧个人家庭的成员关系。
- 提醒订阅授权等账号级设置。
- 旧家庭邀请。

迁移完成后把个人家庭标记为 `archived`，不物理删除，便于失败恢复和审计。

## 云端初始化与配置

### 必要集合

至少确认以下集合存在：

- `user_profiles`
- `families`
- `family_members`
- `family_invites`
- `family_audit_logs`
- `family_settings`
- `user_food_records`

云函数可尝试初始化集合，但上线流程不能只依赖运行时自动创建。部署文档应明确给出云开发控制台的手动创建清单。

### 环境 ID

环境 ID 从微信开发者工具的云开发控制台获取，入口为环境选择器或“设置 / 环境设置”。客户端写入不提交 Git 的 `utils/cloudConfig.local.js`：

```js
module.exports = {
  cloudEnvId: '你的环境ID',
  useCloudFoodApi: true
}
```

账号页联调前应执行一次开发健康检查，确认：

- 当前客户端环境 ID 与部署云函数的环境一致。
- `login`、`accountApi`、`familyApi`、`foodApi` 已部署。
- 必要集合可以由云函数访问。

## 模块边界

- `accountService`：建立账号会话、保存资料、调度后台同步。
- `accountApi`：读取和更新 `user_profiles`，返回明确错误码。
- `familyApi`：家庭状态、邀请、成员权限和个人家庭迁移。
- `foodApi`：按有效家庭读写食材，提供幂等合并。
- 家长账号页：采集头像昵称、展示登录进度，不直接实现迁移。
- 家庭页：展示分享入口和邀请确认，不在前端决定成员资格。

## 安全与权限

- 家庭归属限制由云函数强制执行，不能只隐藏前端按钮。
- 创建者和管理员可邀请成员；只有创建者可修改管理员身份和关键权限。
- 邀请标识使用不可预测随机值，不在日志中完整输出。
- 云函数从调用上下文取得 OpenID，不接受客户端传入的操作者身份。
- 分享卡片只展示邀请人主动设置的昵称和家庭名称。

## 测试与验收

### 登录页面

- 微信头像始终按正方形裁剪展示。
- 选择微信昵称和手动输入都会更新提交值。
- 空昵称不能登录。
- 账号会话成功后立即返回，不等待食材合并。
- 头像上传或同步失败不会丢失本机食材。

### 同步状态

- 集合缺失、函数未部署、环境不一致和断网分别产生正确原因码。
- 用户页面不显示技术报错，也不把非网络错误描述为网络问题。
- 连续点击重试只启动一个同步任务。
- 重复合并不生成重复记录。

### 分享邀请

- 分享卡片携带有效 `inviteId` 并进入邀请确认页。
- 未登录用户登录后能恢复邀请上下文。
- 个人家庭账号可以加入第一个正式家庭。
- 正式家庭账号不能加入另一个家庭。
- 已加入目标家庭时重复打开邀请不会重复创建成员。
- 过期、已用和撤销邀请均有明确提示。

### 家庭状态

- 创建第一条邀请后家庭立即正式化。
- 正式家庭成员减少到一人后仍保持正式状态。
- 旧单人自动家庭可以正确补记为个人家庭。
- 加入迁移失败后可以安全重试，不产生两套有效成员关系。

## 发布顺序

1. 手动确认或创建必要数据库集合。
2. 部署 `login` 和 `accountApi`。
3. 部署包含家庭状态和分享邀请逻辑的 `familyApi`。
4. 部署支持幂等合并的 `foodApi`。
5. 在客户端配置正确环境 ID。
6. 使用两个从未加入正式家庭的微信账号完成真机邀请测试。
7. 再验证已有正式家庭账号无法接受其他邀请。
8. 最后验证断网、集合缺失和重复点击场景。

## 上线验收标准

- 家长账号可在合理时间内完成登录并返回“我的”页。
- 微信头像和昵称交互符合预期。
- 同步失败不影响本机继续记录食材。
- 老人通过微信分享卡片最多两次点击即可加入家庭。
- 任意入口都不能让一个账号同时属于两个正式家庭。
- 开发者可从结构化日志定位集合、部署、环境或网络问题。
