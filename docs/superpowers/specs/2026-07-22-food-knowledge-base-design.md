# 食材知识库、搜索与保存规则设计

更新日期：2026-07-22
状态：已确认设计，待编写实现计划

## 1. 背景

当前项目已经具备约 100 种通用食材、名称/别名搜索、保存提醒、宝宝月龄推荐和 CloudBase `food_base` 集合，但继续扩充前存在三个结构性问题：

1. 本地 `utils/foodBase.js`、扩展食材文件以及多个云函数内存在食材或保存建议副本，容易产生版本差异。
2. 页面搜索主要经由本地 repository 完成，云端新增或修正的数据不一定及时进入实际搜索结果。
3. 食材身份、别名、保存条件、精确期限和证据来源混在同一类数据中，无法稳定完成字段级审核、冲突检测、版本发布与回滚。

本设计将食材数据升级为可审核、可发布、可离线使用的结构化知识库。RAG 或向量检索不是第一阶段核心，只能在后续作为开放式解释和低优先级候选召回能力，不能决定保存期限或宝宝食用安全结论。

## 2. 产品范围

### 2.1 本期范围

- 收录通用食材和常见加工状态，不收录具体品牌、包装规格或商品条码 SKU。
- 同时提升食材搜索召回率和保存建议的准确性、可追溯性。
- 继续使用现有微信 CloudBase 文档数据库。
- 支持弱网和离线搜索；小程序使用已发布知识库快照。
- 精确期限只来自审核通过且有直接证据的规则。
- 保留自定义食材入口，但不为没有直接规则的自定义食材生成精确期限。

### 2.2 非目标

- 不建设品牌商品库、条码库或电商 SKU 同步。
- 不让大模型自动生成并直接发布食材规则。
- 不在第一阶段购买或维护独立向量数据库、Elasticsearch 或 PostgreSQL。
- 不在第一阶段开发完整内容管理后台。
- 不把产品提醒表述为绝对安全期限、医疗判断或包装说明的替代品。

## 3. 成功标准

### 3.1 搜索

- 标准名与已审核别名的 Top 1 命中率为 100%。
- 拼音和已审核常见错别字的 Top 3 命中率不低于 95%。
- 不相关食材被自动选中的数量为 0。
- 搜索无结果时仍允许创建自定义记录，并形成可统计的零结果反馈。

### 3.2 保存规则

- 没有直接证据时生成精确保存天数的数量为 0。
- 上线精确期限规则具备有效来源的比例为 100%，且成人与宝宝期限分别有适用证据。
- 未审核、缺来源、字段不完整或存在同优先级冲突的规则进入正式版本的数量为 0。
- 任一用户记录能够回查当时使用的规则、规则版本、知识库版本和起算时间。

### 3.3 发布可靠性

- 发布失败或快照损坏时，小程序继续使用上一个有效版本。
- 知识库版本切换不覆盖用户的食材记录。
- 普通知识更新不静默改变已有记录的提醒日期。

## 4. 方案选择

### 4.1 采用：CloudBase 分层知识库

将编辑数据、发布元数据、运行快照和用户记录分层。该方案可以沿用当前登录、家庭共享、云函数和本地缓存架构，适合数百至数千种通用食材。

### 4.2 不采用：每种食材一个大文档

单文档方案开发快，但食材、别名、状态规则和来源耦合，难以复用证据、检测规则冲突和执行字段级审核。它不适合后续持续扩充。

### 4.3 暂不采用：关系数据库加独立搜索/向量服务

该方案适合大规模 SKU、复杂管理权限和海量语义查询，但会带来数据迁移、独立服务和运维成本。当前业务规模没有足够收益。

## 5. 总体架构

系统分为四层：

1. **编辑层**：维护食材、搜索词、保存规则和证据来源。
2. **审核发布层**：校验候选数据，生成不可变知识库版本。
3. **运行层**：生成小程序可直接搜索和匹配的扁平快照。
4. **用户层**：保存家庭食材记录及其采用的规则快照。

数据流为：

```text
录入表格/导入文件
  -> 编辑集合
  -> 自动校验
  -> 人工审核
  -> 候选版本
  -> 发布快照
  -> 小程序下载并校验
  -> 本地搜索与规则匹配
  -> 用户选择、保存及反馈
```

CloudBase 是唯一线上事实源。代码仓库仅保留由正式发布版本自动生成的离线快照，不再手工维护多份种子库。

## 6. 数据模型

### 6.1 `foods`：食材身份

保存相对稳定的食材信息，不直接保存所有搜索词和状态规则。

```js
{
  foodId: 'tomato',
  canonicalName: '番茄',
  category: '蔬菜',
  subCategory: '茄果瓜类',
  defaultState: 'raw_whole',
  allergenTags: [],
  riskTags: ['choking_shape_attention'],
  iconKey: 'tomato',
  status: 'active',
  revision: 3,
  reviewStatus: 'approved',
  createdAt: Date,
  updatedAt: Date
}
```

约束：

- `foodId` 全局唯一且发布后不可复用。
- 修改展示名称不能改变 `foodId`。
- 分类使用受控枚举，不接受任意新字符串。
- `status=inactive` 的食材可从新搜索结果中隐藏，但不能删除其历史关联。

### 6.2 `food_search_terms`：可搜索名称

```js
{
  termId: 'tomato-alias-xihongshi',
  foodId: 'tomato',
  term: '西红柿',
  normalizedTerm: '西红柿',
  type: 'alias',
  region: '全国',
  weight: 90,
  reviewStatus: 'approved',
  createdAt: Date,
  updatedAt: Date
}
```

`type` 仅允许：

- `canonical`：标准名称；
- `alias`：稳定同义词；
- `regional`：地域叫法；
- `pinyin`：全拼或首字母；
- `typo`：审核过的常见错别字。

约束：

- 同一 `normalizedTerm` 若指向多个食材，必须标为歧义词并在运行快照中保留多个候选，不得自动选中一个。
- 品种或状态导致规则显著不同的名称应建立独立 `foodId`，不能只作为别名。
- 拼音、首字母等机械字段由发布程序生成，人工数据只维护需要审核的语义关系。

### 6.3 `storage_rules`：条件化保存规则

一条规则只描述一个明确场景。

```js
{
  ruleId: 'tomato-cut-fridge-v2',
  foodId: 'tomato',
  foodState: 'cut',
  storageMethod: 'fridge',
  packageState: 'not_applicable',
  temperatureMinC: 0,
  temperatureMaxC: 4,
  referenceDateType: 'cut_at',
  babyDaysMin: null,
  babyDaysMax: null,
  adultDaysMin: 1,
  adultDaysMax: 2,
  priority: 100,
  advice: '切开后密封冷藏，并尽快食用。',
  discardSigns: ['霉点', '异味', '发黏'],
  evidenceLevel: 'direct',
  evidenceBindings: [
    {
      sourceId: 'source-001',
      audience: 'general',
      locator: '对应食材类别和冷藏条件'
    }
  ],
  reviewStatus: 'approved',
  ruleVersion: 2,
  validFrom: Date,
  validTo: null,
  createdAt: Date,
  updatedAt: Date
}
```

首批 `foodState` 控制为：

- `raw_whole`：完整未处理；
- `washed`：已清洗；
- `cut`：已切开；
- `opened`：包装已开封；
- `cooked`：已烹饪；
- `homemade_baby_food`：自制辅食；
- `thawed`：已解冻。

`storageMethod` 控制为 `room`、`fridge`、`freezer`。`referenceDateType` 必须与场景一致，允许 `purchased_at`、`washed_at`、`cut_at`、`opened_at`、`cooked_at`、`made_at`、`thawed_at`。

`evidenceLevel` 控制为：

- `direct`：来源直接支持该食材、状态和保存条件，可发布有边界的期限；
- `derived`：从同类食材或通用原则推导，只发布定性建议；
- `insufficient`：证据不足，不发布精确天数。

只有 `direct` 规则可以包含期限字段，且每个非空期限必须有适用于对应人群的证据绑定：

- `adultDaysMin/Max` 可以由 `audience=general` 或 `audience=adult` 的直接证据支持；
- `babyDaysMin/Max` 必须由 `audience=baby` 的直接证据支持，不能把成人或一般家庭保存期限自动缩短后伪装成宝宝期限；
- 来源只支持一般保存期限时，宝宝期限字段保持为空，页面可展示一般保存提醒和宝宝谨慎提示，但不得显示独立的“宝宝还能保存几天”；
- `derived` 和 `insufficient` 的所有期限字段必须为空。

### 6.4 `evidence_sources`：证据来源

```js
{
  sourceId: 'source-001',
  organization: 'FoodSafety.gov',
  title: 'Cold Food Storage Chart',
  url: 'https://www.foodsafety.gov/food-safety-charts/cold-food-storage-charts',
  sourceType: 'government_guideline',
  locale: 'en-US',
  publishedAt: Date,
  accessedAt: Date,
  applicableScope: '冷藏和冷冻保存',
  locator: '表格中的具体品类或章节',
  evidenceExcerpt: '不超过审核所需的短摘录或结构化摘要',
  archivedFileId: null,
  status: 'active',
  createdAt: Date,
  updatedAt: Date
}
```

来源与规则分离，以支持一个来源关联多条规则，并在来源更新或失效时找到所有受影响规则。`archivedFileId` 可指向 CloudBase 存储中的授权存档；不得上传或长期保存无权复制的完整受版权保护资料。

现有 [食材库数据来源与安全边界](../../food-data-sources.md) 作为来源优先级和表达边界的基础说明。

### 6.5 `knowledge_releases`：发布版本

```js
{
  releaseId: 'food-kb-2026.08.0',
  status: 'published',
  foodCount: 500,
  searchTermCount: 1200,
  ruleCount: 1800,
  snapshotChecksum: '0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef',
  previousReleaseId: 'food-kb-2026.07.0',
  validationSummary: {
    errors: 0,
    warnings: 3
  },
  createdBy: 'reviewer-id',
  reviewedBy: 'reviewer-id',
  createdAt: Date,
  publishedAt: Date
}
```

`snapshotChecksum` 保存 SHA-256 digest 的小写 hex 文本，长度恰为 64 个字符，不带算法前缀。

版本发布后不可原地修改。任何内容变化均生成新 `releaseId`。

### 6.6 `food_search_docs`：运行快照

该集合由发布程序生成，禁止人工编辑。

```js
{
  foodId: 'tomato',
  canonicalName: '番茄',
  normalizedCanonicalName: '番茄',
  searchTerms: ['番茄', '西红柿', 'fanqie', 'fq'],
  rankedTerms: [
    { term: '番茄', type: 'canonical', weight: 100 },
    { term: '西红柿', type: 'alias', weight: 90 }
  ],
  category: '蔬菜',
  subCategory: '茄果瓜类',
  iconKey: 'tomato',
  activeRuleIds: [
    'tomato-whole-room-v1',
    'tomato-cut-fridge-v2'
  ],
  releaseId: 'food-kb-2026.08.0'
}
```

快照必须同时提供：

- CloudBase 运行集合；
- 代码仓库中的离线 JSON/JS 产物；
- 版本号、生成时间和校验码。

### 6.7 `search_events`：搜索质量反馈

```js
{
  normalizedQuery: '宝宝菠菜泥',
  resultCount: 0,
  topCandidateFoodIds: [],
  selectedFoodId: null,
  action: 'created_custom_food',
  releaseId: 'food-kb-2026.08.0',
  appVersion: '1.2.0',
  createdAt: Date
}
```

仅记录改进搜索所需的数据，不记录宝宝姓名、家庭成员、备注原文、图片或其他无关个人信息。原始查询只在确有必要且通过隐私评估后保存；默认保存标准化查询并设置保留期限。

### 6.8 `user_food_records` 增量字段

现有集合保留，并增加：

```js
{
  foodBaseId: 'tomato',
  foodState: 'cut',
  storageMethod: 'fridge',
  referenceDateType: 'cut_at',
  referenceDate: '2026-07-22',
  appliedRuleId: 'tomato-cut-fridge-v2',
  appliedRuleVersion: 2,
  appliedReleaseId: 'food-kb-2026.08.0',
  adviceSnapshot: {
    babyExpireDate: '2026-07-24',
    adultExpireDate: '2026-07-24',
    advice: '切开后密封冷藏，并尽快食用。',
    computedAt: '2026-07-22T08:00:00+08:00'
  }
}
```

保留现有 `foodBaseId` 字段以降低迁移风险；在新模型中其语义等同于 `foodId`。本期不同时引入第二个重复 ID 字段。

## 7. 搜索设计

### 7.1 标准化

输入依次执行：

1. 去除首尾空白并统一全角/半角字符；
2. 英文字母转小写；
3. 生成中文原文、拼音全拼和首字母查询形式；
4. 保留用户可见原文用于页面展示；
5. 不在标准化阶段静默替换歧义词或错别字。

繁简转换只有在采用经过测试的确定性映射后才启用；未启用时通过审核词条覆盖高频繁体写法。

### 7.2 召回与排序

排序优先级为：

1. 标准名完全一致；
2. 别名完全一致；
3. 标准名前缀；
4. 别名前缀；
5. 拼音全拼或首字母；
6. 审核过的常见错别字；
7. 编辑距离模糊匹配；
8. 用户最近选择和全局选择频率，仅用于同分排序。

高置信度精确名称可作为正常结果置顶。错别字、歧义词和模糊匹配必须显示“你可能在找”，不能静默绑定。

### 7.3 组合输入

“宝宝菠菜泥”等输入可拆为候选食材和状态词：

- `菠菜` -> `foodId=spinach`；
- `泥` -> 候选 `foodState=homemade_baby_food`。

状态识别只用于预填表单，必须由用户确认后才参与规则匹配。

### 7.4 零结果

无合适结果时：

- 提供自定义食材入口；
- 保存 `search_event`；
- 不生成虚假的食材关联；
- 不生成精确保存期限；
- 后续根据零结果频率补充食材、别名或错别字词条。

### 7.5 离线与云端

- 小程序启动或进入食材功能时检查活动版本。
- 新版本下载成功并通过校验后，原子替换本地快照。
- 下载失败时继续使用上一有效快照。
- 首次使用且无法联网时使用随包离线快照。
- 页面输入过程只搜索本地索引，避免逐字请求云函数。

## 8. 保存规则匹配

### 8.1 必要输入

匹配精确规则至少需要：

- `foodBaseId`；
- `foodState`；
- `storageMethod`；
- 与规则一致的 `referenceDateType` 和 `referenceDate`。

包装食品还需记录未开封/已开封状态。用户输入的包装说明和日期优先于通用知识库规则。

### 8.2 匹配优先级

1. 具体食材 + 具体状态 + 具体保存方式 + 可选包装/温度条件；
2. 具体食材 + 具体状态的保守规则；
3. 同一食材的定性通用原则；
4. 同类食材的定性提示；
5. 信息不足。

只有前两级且 `evidenceLevel=direct` 时可以计算精确提醒日期。后三层只能展示原则性提示。

### 8.3 规则具体程度

发布程序按以下确定性分值检测候选规则：

- 指定食材：+100；
- 指定处理状态：+40；
- 指定保存方式：+20；
- 指定包装或自制条件：+10；
- 指定温度范围：+5。

运行时先选择适用且分值最高的规则。同分规则若期限或建议不一致，发布必须失败，不能留到运行时随机解决。

### 8.4 不确定与风险状态

遇到以下情况时停止计算“还能保存几天”：

- 冷链中断或保存过程不确定；
- 起算时间无法确认；
- 已出现霉变、异味、发黏、胀包等异常；
- 包装破损或包装说明与通用规则冲突；
- 没有适用的直接证据规则。

产品应展示保守处理提示，并明确具体包装说明和实际状态优先。

### 8.5 规则更新

- 新记录使用当前正式版本。
- 普通知识更新不改变旧记录的 `adviceSnapshot`。
- 重大安全修正可重新计算仍处于活动状态的记录，但必须显示“保存建议已更新”，并写入审计日志。
- 已完成、已删除或历史记录不自动重算。

## 9. 数据录入、审核与发布

### 9.1 初期录入方式

第一阶段使用固定表格模板维护 `foods`、`food_search_terms`、`storage_rules` 和 `evidence_sources`，导出 CSV/JSON 后由校验和导入程序写入 CloudBase 草稿数据。暂不开发完整管理后台。

### 9.2 状态流转

```text
draft -> sourced -> validated -> approved -> release_candidate -> published
```

任何自动校验或人工审核失败均退回 `draft`。发布版本中的数据不可直接编辑。

小团队初期允许同一账号承担录入和发布，但系统仍分别记录 `createdBy`、`reviewedBy` 和 `publishedAt`；涉及精确期限和宝宝高风险提示时，发布前必须执行独立的内容复核清单。

### 9.3 自动校验

发布前至少检查：

- Schema 和枚举合法；
- ID 唯一；
- 外键引用存在；
- 标准名、别名和歧义词冲突；
- `direct` 规则的每个期限字段都有适用于对应人群的有效来源；
- `derived/insufficient` 规则不包含期限；
- 起算时间与处理状态一致；
- 同具体程度规则不存在冲突；
- 所有运行快照记录均来自同一候选版本；
- 生成结果可复现且校验码稳定。

### 9.4 发布和回滚

发布顺序：

1. 从审核通过的数据生成完整新快照；
2. 执行结构、关联、冲突和内容边界校验；
3. 计算快照校验码；
4. 写入 `release_candidate`；
5. 完成发布审核；
6. 原子切换活动 `releaseId`；
7. 保留上一版本供即时回滚。

任何步骤失败均不得改变活动版本。回滚通过切换活动版本完成，不批量覆盖用户记录。

## 10. 迁移方案

### 10.1 基线快照

将当前正式可用食材库导出为 `legacy-v1`，记录数量、ID、别名、规则和校验码，作为迁移比较和紧急回退依据。

### 10.2 迁移编辑数据

1. 现有食材导入 `foods` 草稿。
2. 名称和别名导入 `food_search_terms`。
3. 检查错误合并与歧义关系，例如小番茄/圣女果与番茄、香菇与蘑菇。
4. 现有保存范围转换成 `storage_rules`，初始标记为 `legacy_unverified` 内部迁移状态。
5. 逐条完成来源审核后转为 `direct`、`derived` 或 `insufficient`。

`legacy_unverified` 只用于迁移工作区，不能进入正式候选版本。

### 10.3 双轨比较

迁移期间并行执行旧搜索和新搜索，但只向用户展示旧结果，记录以下差异：

- 旧系统命中、新系统未命中；
- 新旧第一候选不同；
- 新系统把一个旧别名识别为歧义；
- 新系统没有精确规则而旧系统给出期限。

差异必须完成审核或明确接受后才能切换搜索入口。

### 10.4 分步切换

1. 先切换食材列表和搜索快照读取。
2. 再为新记录启用状态化规则匹配。
3. 旧记录继续使用原有字段和计算结果。
4. 验证稳定后，逐步移除人工维护的云函数种子副本。
5. 最终仅保留发布程序生成的随包离线快照。

切换使用功能开关控制，能够分别回退搜索和规则匹配，不要求整体回滚应用版本。

## 11. API 边界

`foodApi` 建议形成以下只读能力：

- `getKnowledgeManifest`：返回活动版本、上一版本、校验码和生成时间；
- `getKnowledgeSnapshot`：按版本返回完整快照或分页资源地址；
- `getFoodById`：按稳定 `foodId` 读取已发布食材；
- `getStorageRule`：仅供服务端诊断或必要回退，正常页面优先使用本地正式快照；
- `reportSearchEvent`：写入最小化搜索反馈。

发布、审核和导入能力使用独立管理云函数或离线发布工具，不复用面向小程序用户的 `foodApi`，并限制为管理员身份。

## 12. 权限、隐私与安全

- 编辑集合和发布操作只允许管理端访问。
- 小程序用户只能读取已发布运行快照。
- `search_events` 仅允许云函数写入，客户端不能直接修改统计结果。
- 查询日志不关联宝宝姓名、生日、家庭备注或图片。
- 来源摘录保留审核所需最小内容，并记录原始链接和定位信息。
- 所有精确规则输出附带非绝对安全期限提示；包装说明、实际状态和专业意见优先。
- RAG 后续只能读取已发布证据，不具备写回正式知识库或修改规则的权限。

## 13. 缓存与故障处理

- 本地保存当前和上一份有效快照。
- 新快照校验失败时丢弃新文件，继续使用当前版本。
- manifest 请求失败不阻塞搜索和添加记录。
- 规则匹配异常时退化为定性提示，不生成默认精确天数。
- 食材存在但图标缺失时使用无语义通用图或隐藏图标，不能借用其他具体食材图。
- 搜索反馈上传失败不影响用户保存食材。
- 发布工具中途失败时保留候选数据，但不改变活动版本。

## 14. 测试策略

### 14.1 数据校验测试

- Schema、枚举、必填字段和 ID 唯一性；
- 食材、搜索词、规则和来源之间的引用完整性；
- 歧义词与错误合并检测；
- 证据等级、适用人群与期限字段一致性；
- 同优先级规则冲突检测；
- 快照生成确定性和校验码稳定性。

### 14.2 搜索黄金集

为标准名、别名、地域词、拼音、首字母、错别字、歧义词、组合输入和无结果查询建立固定用例。测试 Top 1、Top 3、候选标签和是否发生自动选中。

### 14.3 规则黄金集

覆盖每种处理状态和保存方式，验证：

- 选择了正确的最具体规则；
- 起算时间正确；
- `derived/insufficient` 不产生日期；
- 包装日期优先；
- 不确定保存过程停止倒计时；
- 重大安全更新只影响符合条件的活动记录。

### 14.4 迁移与兼容测试

- 现有用户记录可继续读取、编辑和完成；
- `foodBaseId` 在新旧版本中保持稳定；
- 搜索和规则功能开关可独立回退；
- 云端不可用时随包快照可用；
- 新快照损坏时仍能加载上一版本；
- 当前家庭共享、提醒和本地登录模式不因知识库切换而改变。

### 14.5 真机验证

自动测试通过后仍需在微信开发者工具和真机验证：

- 首次安装、升级和清缓存后的快照下载；
- 弱网、断网和版本下载中断；
- 输入响应速度和中文输入法组合态；
- 新旧版本切换后的搜索结果与用户记录；
- 云函数重新部署后的 manifest 和权限。

## 15. 分阶段交付

### 阶段 A：统一底座

- 落地集合 Schema、校验器和发布 manifest。
- 迁移并审核现有约 100 种食材。
- 生成第一份正式运行快照。
- 接通本地搜索，消除手工维护的食材副本。

### 阶段 B：搜索与状态化规则

- 完成分层排序、拼音、错别字、歧义提示和零结果反馈。
- 添加状态、保存方式和起算时间表单。
- 新记录保存规则与知识库版本快照。

### 阶段 C：扩充与反馈闭环

- 根据高频食材和真实零结果分批扩充到约 300 至 500 种通用食材。
- 优先提高高频食材的别名覆盖和直接证据规则覆盖。
- 建立定期来源复核、失效检查和版本发布节奏。

### 阶段 D：可选智能检索与问答

只有在自然语言复合查询量足够、黄金测试集建立且前三阶段稳定后，才评估：

- 关键词与向量的混合候选召回；
- 基于已发布证据的 RAG 解释；
- 答案来源展示与无法回答时的明确拒答。

向量或 RAG 结果不能覆盖精确名称命中、结构化规则或包装说明。
