# 家庭成员退出与移出设计

## 目标

让家庭共享组具备完整的成员退出机制：家庭创建者可以移出其他成员，普通成员和管理员可以主动退出，同时保留家庭食材与操作历史。

## 权限规则

- 只有家庭创建者可以移出成员。
- 创建者只能移出普通成员或管理员，不能移出自己。
- 普通成员和管理员可以主动退出当前家庭。
- 创建者不能直接退出。后续如需支持，应先实现转让创建者或解散家庭，本次不扩展。

## 数据行为

- 退出或移出采用软删除，将对应 `family_members` 记录的 `status` 更新为 `inactive`。
- 家庭中的食材、采购计划及历史操作记录继续保留，不随成员关系删除。
- 操作写入 `family_audit_logs`，记录操作者、目标成员、家庭、动作类型和时间。
- 退出或被移出后，账号再次访问家庭功能时自动获得新的个人家庭，可继续保存自己的食材，也可以接受其他家庭邀请。

## 页面交互

### 成员管理页

- 创建者在每个非创建者成员行看到“移出家庭”按钮。
- 点击后显示二次确认，明确成员将无法继续查看和管理该家庭食材，但历史记录保留。
- 操作成功后刷新成员列表并提示“已移出家庭”。
- 操作失败时保留页面状态，并展示服务端返回的可理解错误。

### 家庭共享页

- 普通成员和管理员在页面底部看到“退出家庭组”。
- 创建者不显示退出按钮。
- 点击后显示二次确认，明确退出后无法继续访问该家庭食材。
- 操作成功后重新加载家庭数据，展示账号新建的个人家庭。
- 操作失败时保留当前家庭并展示错误。

## 服务接口

- `removeMember({ openId })`：要求当前账号拥有 `manage_members` 权限，目标必须是同一家庭的活跃非创建者成员。
- `leaveFamily()`：要求当前账号是活跃的普通成员或管理员，创建者调用时返回明确错误。
- 两个接口保持幂等保护：目标关系不存在或已经失效时返回明确错误，不修改其他记录。

## 测试范围

- 创建者可以移出普通成员和管理员。
- 普通成员、管理员和非成员不能移出他人。
- 创建者不能移出自己，也不能主动退出。
- 普通成员和管理员可以退出，成员关系变为失效。
- 退出或移出后历史家庭数据仍存在，账号可获得新的个人家庭。
- 页面仅为正确身份显示操作入口，确认、取消、成功和失败流程均有覆盖。
